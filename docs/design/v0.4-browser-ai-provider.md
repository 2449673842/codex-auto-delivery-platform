# v0.4 Browser AI Provider Design — Multi-AI Collaboration & Token Budget Strategy

> 设计文档：多 AI 协作 + Token 预算策略
> 本阶段只做设计，不实现。

---

## 1. Design Goals

多 AI 协作不是让所有 AI 都看完整上下文，而是让每个 AI 只看它完成任务所需的最小上下文。

Goals:

- Reduce token consumption
- Avoid repeatedly sending fixed project context
- Avoid sending entire repo to AI
- Avoid multiple AIs doing the same work
- Let cheap models handle simple tasks
- Let strong models handle only high-value / high-difficulty steps
- Let Browser AI Reviewer provide read-only advisory review
- All AI outputs converge into unified AgentRunResult / Governance / Patch Sandbox / Sandbox Gate

## 2. Reasonix-inspired Design Principles

Reference only — do not copy code.

### A. Stable Project Prefix

Keep stable project context as a fixed prefix:

- Project goal
- Current version
- AGENTS.md summary
- Safety boundaries
- Output format requirements
- Current capability scope
- Known non-goals

This prefix should be byte-stable / content-stable to enable cache reuse by future API providers.

Do not regenerate a different version of the project intro each time.

### B. Append-only Task Log

Each task's factual record uses append-only:

- User requirement
- AI plan
- Patch summary
- Sandbox result
- Governance result
- Review packet
- Reviewer opinion
- Human decision

Do not frequently rewrite history. Generate a compact summary when needed, but keep original records as artifacts.

### C. Volatile Scratch excluded from long-term context

AI intermediate thoughts, temporary reasoning, and drafts should not enter long-term context.

Only save structured results:

- `plan.md`
- `patch.diff`
- `review.md`
- `risk_report.json`
- `review_packet.json`
- `browser_ai_review.json`

### D. Turn-end / Step-end Compaction

Large outputs must not be carried into subsequent prompts indefinitely.

Examples:

- Long PR diff → `diff_summary`
- Long Sonar comments → `sonar_summary`
- Long pytest output → `test_summary`
- Long file content → relevant snippets
- Long browser response → parsed artifact + summary

Subsequent AIs consume the summary by default. Fetch raw content on demand when needed.

### E. Cheap-first / Strong-on-demand

Default to cheap models for:

- Project summary
- Task classification
- Context compression
- Risk precheck
- Review packet explanation
- Simple review

Escalate to strong models only for:

- Patch generation
- Complex architecture planning
- Repeated malformed output
- Multiple sandbox apply failures
- High-risk governance needing explanation
- Low reviewer confidence
- Human_required situations needing advisory

### F. Failure-triggered Escalation

Escalate on these signals:

- `malformed_response`
- `invalid_patch_diff`
- `context_mismatch`
- `output_not_diff`
- Repeated governance failure
- Review conflict
- Low extraction confidence
- Browser AI response incomplete

Escalation must be explicitly logged. Silent cost increase is forbidden.

## 3. Multi-AI Role Assignment

Design only — not implemented in this phase.

### 3.1 Planner AI

**Input:**
- Project Brief
- Task description
- Allowed files
- Relevant code summary

**Output:**
- `plan.md`

**Recommended model:** cheap / flash / browser AI

### 3.2 Coder AI

**Input:**
- `plan.md`
- Minimal code context
- `allowed_files`
- `forbidden_files`
- Output contract

**Output:**
- `patch.diff`

**Recommended model:** API strong model or Browser AI Coder

**Restrictions:**
- Output unified diff only
- No explanations
- No real repo writes
- No PR creation

### 3.3 Reviewer AI

**Input:**
- `patch.diff`
- Changed files summary
- Governance summary
- Sandbox result

**Output:**
- `review.md`

**Responsibilities:**
- Check for out-of-scope modifications
- Check test quality
- Check safety boundaries
- Check for relaxed assertions
- Check for empty tests / pass placeholders

### 3.4 Risk AI

**Input:**
- Patch summary
- File path list
- Governance findings
- Review findings

**Output:**
- `risk_report.json` (JSON schema: risk_level, requires_human, reasons, security_findings, scope_findings, test_findings)

### 3.5 Browser AI Reviewer

**Input:**
- Review Packet
- PR body
- Changed files
- Sonar summary
- Test summary
- AGENTS.md summary

**Output:**
- `browser_ai_review.json` (advisory only)

**Restrictions:**
- Read-only
- No comments
- No approve
- No merge
- No PR body edits
- No CI trigger
- Not final merge criteria

### 3.6 Rule Gate

Not an LLM. Includes:

- AI Output Governance
- Patch Sandbox
- Sandbox Gate
- Review Packet rules
- ApprovalPolicy

**Responsibilities:**
- Hard rule enforcement
- LLM cannot bypass rule gate
- `human_required` cannot be auto-approved by AI

## 4. Token Budget Policy

### Budget fields

```json
{
  "max_context_tokens": 12000,
  "max_code_context_tokens": 6000,
  "max_diff_tokens": 4000,
  "max_review_packet_tokens": 3000,
  "max_browser_prompt_tokens": 8000,
  "max_response_tokens": 4000,
  "prefer_summary_over_raw": true,
  "allow_strong_model_escalation": true
}
```

### Per-mode budgets

| Mode | Default context | Strong model allowed |
|------|----------------|---------------------|
| planning | project brief + task + file summary | Optional |
| patch_generation | plan + selected files only | Yes |
| review | patch + summaries | Optional |
| risk | summary only | No, default cheap |
| browser_reviewer | review packet + summary | Optional |
| repair | failed artifact + error summary | Yes |

## 5. Context Packet Layering

Browser AI / API AI should not consume raw full context.

### Layer 1: Project Stable Prefix

Stable content:

- Project goal
- Current version
- AGENTS.md summary
- Safety boundaries
- Output contracts
- Current implemented modules
- Known non-goals

### Layer 2: Task Context

Task-specific:

- Task title
- Task description
- Current status
- Allowed files
- Forbidden files
- Expected artifacts

### Layer 3: Relevant Code Context

Only relevant files:

- Schema
- Service
- Router
- Tests
- Frontend page if needed

Large files must be truncated / summarized.

### Layer 4: Runtime Evidence

Run results:

- pytest summary
- compileall summary
- npm build summary
- Sonar summary
- Sandbox result
- Review packet
- Governance finding

Default to summary, not full raw content.

Do not ad-hoc assemble prompts each time.

- `project_brief_v1`
- `planning_prompt_v1`
- `patch_generation_prompt_v1`
- `review_prompt_v1`
- `risk_prompt_v1`
- `browser_reviewer_prompt_v1`

- `summary_prompt_v1`

### Template metadata

- `purpose`
- `input_fields`
- `output_contract`
- `max_tokens`
- `allowed_model_tiers`
- `safety_notes`

This enables stable, auditable, cacheable prompts.

## 8. Multi-AI Collaboration Flow

Recommended flow:

```
Task created
→ Context Selector selects minimal relevant context
→ Planner AI generates plan.md
→ Coder AI generates patch.diff (based on plan + selected files)
→ Governance validates output
→ Patch Sandbox applies patch
→ Risk AI generates risk_report.json
→ Reviewer AI generates review.md
→ Review Packet Preview API generates review_packet
→ Browser AI Reviewer reads review_packet → advisory browser_ai_review.json
→ Sandbox Gate / ApprovalPolicy → final hard-rule decision
→ Human Approval
```

### Rules

- AIs collaborate via **artifacts**, not shared infinite conversation history
- Each AI consumes only the previous stage's structured artifact
- Large content → summary + hash
- Raw content stays in artifact, fetched on demand
- Rule Gate takes precedence over AI suggestions

## 9. Cost-aware Model Routing

```json
{
  "planner": "cheap",
  "coder": "strong_if_patch_complex",
  "reviewer": "cheap_or_browser",
  "risk": "cheap_structured",
  "summarizer": "cheap",
  "browser_reviewer": "browser_or_cheap",
  "repair": "strong_on_failure"
}
```

### Rules

- Default cheap-first
- Strong model must have a trigger reason
- Each escalation logged with reason
- No silent escalation
- No multiple strong models processing the same material
- Browser AI as cost supplement, not stability guarantee

## 10. Token-saving Rules

- Do not send entire repo
- Do not send full chat history
- Do not send full PR page HTML
- Do not repeatedly send full README / AGENTS — use summary + hash
- Do not send full pytest log to AI — only summary
- Do not send full Sonar page to AI — only QG / Hotspots / Duplication / Issues summary
- Do not send full diff to all AIs — only Coder / Reviewer need diff
- Do not let multiple AIs read the same long file
- Do not send secrets to any AI
- Do not save volatile scratch into artifact
- Large artifacts default to compressed summary
- Fetch raw content on demand

---

## 11. Token Budget Guard

Behavior when context exceeds budget:

1. Remove non-relevant files first
2. Compress history next
3. Compress test logs next
4. Compress Sonar / PR comments next
5. Truncate large files next
6. If still over limit → stop, return `context_too_large`
7. Never force-feed oversized context to AI

## 14. Relationship with Browser AI Reviewer

Browser AI Reviewer should consume Review Packet first, not browse the full GitHub PR page.

Recommended flow:

```
GitHub PR
→ Review Packet Preview API
→ compact review_packet.json
→ Browser AI Reviewer
→ advisory browser_ai_review.json
→ Mastermind final review
```

This saves tokens, is stable, and is auditable.

Only when Review Packet is insufficient should Browser AI read GitHub visible page fragments.
