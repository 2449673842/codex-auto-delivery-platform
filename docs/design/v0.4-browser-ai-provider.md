# v0.4 S6 — Browser AI Provider Design

> Design document for Browser AI Provider, Multi-AI Collaboration, and Token Budget Strategy.
> This phase is design only — no implementation.

---

## 1. Browser AI Provider Overview

### Positioning

- **Experimental provider** — not the default provider
- **User-authorized provider** — requires explicit user consent for browser automation
- **Browser-controlled provider** — operates through visible web UI automation
- **Not a replacement** for official API-based providers (SandboxProvider, OpenAI)
- **Not for bypassing** rate limits, CAPTCHA, 2FA, paid-service rules, or terms of service

### Supported Targets (design only)

- ChatGPT Web
- Claude Web
- Gemini Web
- Additional web AI targets — future extension

### Constraints

- All browser AI output must pass through Governance → Patch Sandbox → Sandbox Gate
- Malformed / non-parseable responses → `malformed_response` → AgentRun failed
- No real PR creation, no real CI/Sonar/Deploy calls
- No bypass of project safety boundaries

---

## 2. Browser Provider Modes

### 2.1 Safe Platform Mode (default)

- Encrypted browser session / profile reference only
- No plaintext cookies stored
- No plaintext localStorage stored
- No plaintext credentials (username / password) stored
- No CAPTCHA / 2FA / rate limit bypass
- No hidden internal API calls as primary implementation path
- Automation through visible web UI only
- All output enters Governance → Patch Sandbox → Sandbox Gate

### 2.2 Local Power User Mode

- For user's local machine or self-hosted deployment
- Allows reuse of local browser profile
- Cookies / localStorage remain in the user's local browser profile
- Platform backend does **not** upload, read, or record plaintext cookies
- Session expiry → prompt user to re-login
- Reduces per-login cost for frequent users

### 2.3 Lab Mode

- Internal research mode for evaluating hidden API feasibility
- **Default off** — must be explicitly enabled
- Not part of mainline implementation
- Not part of CI / release acceptance criteria
- Not for bypassing login, paywalls, rate limits, CAPTCHA, risk controls, or service restrictions

## 4. Browser Worker Architecture

### Modules (design only)

| Module | Responsibility |
|--------|---------------|
| **BrowserAIProvider** | Top-level provider implementing AgentProvider interface |
| **BrowserSessionWorker** | Manages individual browser automation sessions |
| **BrowserSessionRegistry** | Registry of known sessions (encrypted) |
| **PersistentSessionStore** | Encrypted session persistence |
| **PromptRedactor** | Redacts secrets from prompts before sending |
| **PromptInjector** | Injects project context into web AI prompts |
| **VisibleResponseExtractor** | Extracts AI response from visible web UI |
| **ResponseParser** | Parses extracted text into AgentRunResult |
| **BrowserAuditLogger** | Logs browser operations for audit |
| **BrowserFailureClassifier** | Classifies failures (network, login, CAPTCHA, malformed, etc.) |

### Flow

```
AgentRun dispatch
→ BrowserAIProvider.select_mode()
→ BrowserSessionWorker.start_or_resume(encrypted_ref)
→ PromptInjector.inject(prompt, context_packet)
→ VisibleResponseExtractor.extract(page)
→ ResponseParser.parse(text) → AgentRunResult
→ Governance.validate(result)
→ PatchSandbox.apply(result)
```

## 5. Browser AI Context Packet & Question Strategy

### 5.1 BrowserAIContextPacket

| Field | Description |
|-------|-------------|
| project_name | Codex Automation Delivery Platform |
| project_goal | AI自动代码交付平台 |
| current_version | v0.4.0 |
| current_completed_scope | v0.1–v0.4 S4 |
| current_task_goal | Task-specific goal |
| allowed_files | Files the AI may modify |
| forbidden_files | Files the AI must NOT modify |
| relevant_code_context | Minimal relevant code |
| relevant_api_context | Relevant API endpoints |
| safety_boundaries | Current safety rules |
| output_contract | Expected output format |
| expected_artifacts | plan.md / patch.diff / review.md / risk_report.json |
| known_risks | Pre-identified risks |
| do_not_do | Explicitly prohibited actions |

### 5.2 Project Brief Prompt

Fixed prompt preamble for every Browser AI interaction:

```
You are assisting the Codex Automation Delivery Platform (v0.4.0).

Current capabilities:
- Project / Task management with 9-state state machine
- AgentProfile / AgentRun / AgentReview
- ApprovalPolicy / ApprovalDecision / RiskAssessment
- AI Provider Sandbox (simulated) + OpenAI Minimal Adapter
- AI Output Governance (validation, secret redaction, forbidden paths)
- Code Context API
- Patch Apply Sandbox
- Sandbox Approval Gate
- Review Packet Preview API (MockGitHub/MockSonar)

Not yet implemented:
- Real GitHub PR creation
- Real CI/Sonar API integration
- Browser AI Provider (this session)
- Deployment automation

Safety boundaries:
- No access to Project.root_path
- No shell / subprocess execution
- No git clone / commit / push
- No real GitHub PR / CI / Sonar / Deploy calls
- No secret_ref reading
- No auto-approve of human_required tasks

All output must:
- Return to AgentRunResult format
- Pass through Governance validation
- Pass through Patch Sandbox
- Pass through Sandbox Gate
```

### 5.3 Question Strategy — 6 Prompt Templates

#### A. Planning Prompt

Goal: Generate `plan.md`.

Requirements:
- Minimal safe implementation approach
- Do not break safety boundaries
- List affected files
- List test plan
- List risk points
- Output: Markdown

#### B. Patch Generation Prompt

Goal: Generate `patch.diff`.

Requirements:
- Output unified diff only
- Modify allowed files only
- No explanatory text outside diff
- No new real external API calls
- No secret reading
- No Project.root_path access
- No shell execution

#### C. Review Prompt

Goal: Generate `review.md`.

Checks:
- Out-of-scope modifications
- Empty tests / pass placeholders
- Relaxed assertions
- Safety boundary violations
- Real external API calls
- Missing tests
- PR body inconsistency with implementation

#### D. Risk Prompt

Goal: Generate `risk_report.json`.

Output format (JSON only):

```json
{
  "risk_level": "low | medium | high | critical",
  "requires_human": true,
  "reasons": [],
  "security_findings": [],
  "scope_findings": [],
  "test_findings": []
}
```

#### E. Test Suggestion Prompt

Goal: Suggest tests.

Requirements:
- Cover success paths
- Cover failure paths
- Cover safety boundaries
- No empty tests
- No `assert True`
- No test-only-title-exists patterns

#### F. PR Review Packet Prompt

Goal: Second review opinion based on Review Packet.

Requirements:
- Use GitHub / Sonar / pytest real data
- Code wins over report if inconsistent
- Output: blockers / warnings / required_actions

### 5.4 What Not To Ask

Do **not** ask the web AI:

- How to bypass CAPTCHA
- How to bypass login
- How to call hidden APIs to avoid restrictions
- How to auto-harvest cookies
- How to read secrets
- How to directly push / merge / deploy
- How to skip tests
- How to pass Sonar without fixing issues
- How to directly access Project.root_path

### 5.5 Output Contract

Browser AI output MUST be parseable into:

- `AgentRunResult`
- `plan.md`
- `patch.diff`
- `review.md`
- `risk_report.json`

If output does not conform:

- Mark `malformed_response`
- AgentRun → failed
- No artifacts created
- Return `human_required` or `agent_failed`

## 7. Browser AI Reviewer Mode

### 7.1 Positioning

Browser AI Reviewer is a **read-only** reviewer.

Goals:
- Read GitHub PR page or Review Packet
- Understand PR body, changed files, diff, Sonar comments, test results
- Output blockers / warnings / required_actions
- Help mastermind reduce manual review effort
- Do **not** approve
- Do **not** merge
- Do **not** comment
- Do **not** modify code

### 7.2 Input Sources

- GitHub PR page
- PR body
- Changed files
- Diff
- GitHub issue comments
- SonarCloud bot comments
- Review Packet Preview API output
- AI1 report
- Release notes / AGENTS.md / harness docs

**First phase priority:**
1. Review Packet Preview API output
2. GitHub PR body
3. Changed files list
4. Sonar bot comment

First phase should **not** have Browser AI browse the full GitHub PR UI independently.

Fixed prompt producing structured JSON output:

```json

  "review_status": "approved | blocked | needs_update",
  "merge_allowed_recommendation": true,
  "blockers": [],
  "warnings": [],
  "required_actions": [],
  "confidence": 0.0,
  "reasoning_summary": ""

```

### 7.5 Output Integration

Reviewer output does **not** directly change Task status.

Output saved as:
- `review.md`
- `browser_ai_review.json`
- Or AgentReview record

Must be tagged:
- `source: browser_ai_reviewer`
- `advisory_only: true`
- `not_final_approval: true`

Final merge still requires mastermind review.

### 7.6 Reviewer Safety Boundaries

**Forbidden:**
- Click Merge
- Click Approve
- Submit GitHub comment
- Edit PR body
- Trigger CI
- Create new PR
- Download secrets
- Open unknown external links
- Read plaintext cookies
- Bypass login
- Call hidden APIs
- Serve as final merge criteria

## 8. Multi-AI Collaboration

> Multi-AI collaboration is not about all AIs seeing the full context.
> Each AI sees only the minimum context needed for its task.

### 8.1 Design Principles (Reasonix-inspired)

Reference only — do not copy code.

#### A. Stable Project Prefix

Keep stable project context as a fixed prefix:

- Project goal
- Current version
- AGENTS.md summary
- Safety boundaries
- Output format requirements
- Current capability scope
- Known non-goals

This prefix should be byte-stable / content-stable to enable cache reuse.

#### B. Append-only Task Log

Each task's factual record uses append-only:

- User requirement → AI plan → patch summary → sandbox result → governance result → review packet → reviewer opinion → human decision

Do not rewrite history. Generate compact summary when needed, but keep originals as artifacts.

#### C. Volatile Scratch excluded from context

AI intermediate thoughts, drafts, and temporary reasoning must not enter long-term context.

Only save structured results:
- `plan.md`
- `patch.diff`
- `review.md`
- `risk_report.json`
- `review_packet.json`
- `browser_ai_review.json`

#### D. Turn-end / Step-end Compaction

Large outputs must not persist into subsequent prompts:

- Long diff → `diff_summary`
- Long Sonar → `sonar_summary`
- Long pytest → `test_summary`
- Long files → relevant snippets
- Long browser response → parsed artifact + summary

Subsequent AIs consume summary by default. Fetch raw content on demand.

#### E. Cheap-first / Strong-on-demand

Default cheap models for: project summary, classification, compression, precheck, simple review.

Escalate to strong models only for: patch generation, complex planning, repeated malformed output, low confidence.

#### F. Failure-triggered Escalation

Escalate on: `malformed_response`, `invalid_patch_diff`, `context_mismatch`, repeated governance failure, low extraction confidence.

Escalation must be explicitly logged. Silent cost increase is forbidden.

### 8.2 AI Role Assignment

| Role | Input | Output | Default model tier |
|------|-------|--------|-------------------|
| Planner | Project brief + task + file summary | `plan.md` | cheap |
| Coder | plan + selected files | `patch.diff` | strong_if_complex |
| Reviewer | patch + summaries | `review.md` | cheap_or_browser |
| Risk | summary only | `risk_report.json` | cheap_structured |
| Browser AI Reviewer | review packet + summary | `browser_ai_review.json` | browser_or_cheap |
| Repair | failed artifact + error summary | fixed output | strong_on_failure |
| Summarizer | large artifacts | compact summary | cheap |

### 8.3 Rule Gate (not LLM)

Hard-rule enforcement, not AI:

- AI Output Governance
- Patch Sandbox
- Sandbox Gate
- Review Packet rules
- ApprovalPolicy

LLM cannot bypass rule gate. `human_required` cannot be auto-approved by AI.

### 8.4 Collaboration Flow

```
Task created
→ Context Selector selects minimal context
→ Planner AI → plan.md
→ Coder AI → patch.diff (plan + selected files)
→ Governance validates output
→ Patch Sandbox applies patch
→ Risk AI → risk_report.json
→ Reviewer AI → review.md
→ Review Packet Preview API → review_packet
→ Browser AI Reviewer → advisory browser_ai_review.json
→ Sandbox Gate / ApprovalPolicy → hard-rule decision
→ Human Approval
```

**Rules:**
- AIs collaborate via artifacts, not infinite conversation history
- Each AI consumes only previous stage's structured artifact
- Large content → summary + hash; raw content on demand
- Rule Gate > AI suggestions

## 9. Token Budget Strategy

### 9.1 Budget fields

```json
{
  "max_context_tokens": 12000,
  "max_code_context_tokens": 6000,
  "max_diff_tokens": 4000,
  "max_review_packet_tokens": 3000,
  "max_browser_prompt_tokens": 8000,
  "max_response_tokens": 4000,
  "prefer_summary_over_raw": true,
  "allow_strong_model_escalation": true
}
```

### 9.2 Per-mode budgets

| Mode | Default context | Strong model allowed |
|------|----------------|---------------------|
| planning | project brief + task + file summary | Optional |
| patch_generation | plan + selected files only | Yes |
| review | patch + summaries | Optional |
| risk | summary only | No, default cheap |
| browser_reviewer | review packet + summary | Optional |
| repair | failed artifact + error summary | Yes |

### 9.3 Context Packet Layering

| Layer | Content | Token weight |
|-------|---------|-------------|
| Layer 1: Project Stable Prefix | goal, version, AGENTS summary, safety, scope | Stable, cached |
| Layer 2: Task Context | title, description, status, allowed/forbidden files | Per task |
| Layer 3: Relevant Code | schema, service, router, tests (truncated) | Per task |
| Layer 4: Runtime Evidence | pytest/compileall/npm/Sonar/sandbox summaries | Per run |

### 9.4 Context Hash / Cache Key

Fields for caching and audit:

- `project_prefix_hash`
- `agents_md_hash`
- `task_context_hash`
- `code_context_hash`
- `review_packet_hash`
- `prompt_template_id`
- `prompt_hash`
- `response_hash`
- `cache_eligible`
- `cache_hit_estimate`
- `token_budget_used`
- `model_tier`

Must NOT contain plaintext secrets.

### 9.5 Prompt Template Registry

Fixed templates:

- `project_brief_v1`
- `planning_prompt_v1`
- `patch_generation_prompt_v1`
- `review_prompt_v1`
- `risk_prompt_v1`
- `browser_reviewer_prompt_v1`
- `repair_prompt_v1`
- `summary_prompt_v1`

Each template has: `template_id`, `purpose`, `input_fields`, `output_contract`, `max_tokens`, `allowed_model_tiers`, `safety_notes`.

### 9.6 Cost-aware Model Routing

```json
{
  "planner": "cheap",
  "coder": "strong_if_patch_complex",
  "reviewer": "cheap_or_browser",
  "risk": "cheap_structured",
  "summarizer": "cheap",
  "browser_reviewer": "browser_or_cheap",
  "repair": "strong_on_failure"
}
```

- Default cheap-first
- Strong model must have logged trigger reason
- No silent escalation
- No multiple strong models on same material

### 9.7 Token-saving Rules

- No entire repo sent
- No full chat history
- No full PR page HTML
- No full README/AGENTS — use summary + hash
- No full pytest log — use summary
- No full Sonar page — use QG / Hotspots / Duplication / Issues
- No full diff to all AIs — only Coder/Reviewer need diff
- No multiple AIs reading the same long file
- No secrets sent to any AI
- No volatile scratch in artifacts
- Large artifacts → compressed summary; raw on demand

### 9.8 Token Budget Guard

When context exceeds budget:

1. Remove non-relevant files
2. Compress history
3. Compress test logs
4. Compress Sonar / PR comments
5. Truncate large files
6. If still over limit → stop, return `context_too_large`
7. Never force-feed oversized context to AI
