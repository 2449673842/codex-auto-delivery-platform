# v0.2 — Playwright 测试结果接入平台

> 设计文档 · 2025-07-16
> 基线 commit：`4a1220462fc4c2e0409d11cb8804f45609d907fb`

---

## 1. v0.2 目标

将 Playwright MCP / 浏览器验收结果**结构化接入平台**，使测试结果成为任务交付流程的一环。

### 核心能力

| 能力 | 说明 |
|------|------|
| **结果接入** | 支持 Playwright MCP、人工、CI 三种 runner 的测试结果提交 |
| **测试详情** | TaskDetail 页面内嵌 Test Run 面板，查看测试状态、耗时、失败步骤 |
| **产物关联** | 测试日志、浏览器控制台错误、Network 失败、截图/trace 直接与 TaskArtifact 关联 |
| **影响决策** | 测试通过/失败作为 Review 的辅助参考信息 |
| **审计追踪** | 所有 TestRun 创建、状态变更、产物上传写入 TaskEvent |

### 本阶段不做

- 不自动启动 Playwright 浏览器
- 不接入 CI（Jenkins / GitHub Actions）
- 不接入 SonarQube
- 不接入 GitHub PR
- 不实现自动返工循环（changes_requested 触发重测）
- 不实现 Test Worker 调度

---

## 2. 功能范围

### 包含（In Scope）

| 模块 | 功能 |
|------|------|
| **TestRun 管理** | 创建、查询、更新状态 |
| **测试产物关联** | 通过 TaskArtifact 存储 playwright_report、截图、console log、network log |
| **TaskDetail 面板** | TestRun 区域展示最近测试结果 |
| **TaskEvent 联动** | 所有 TestRun 操作写入审计日志 |
| **安全约束** | archived Task 禁止新增 TestRun |

### 不包含（Not in v0.2）

| 功能 | 原因 |
|------|------|
| 自动执行 Playwright | 需要沙箱/worker 进程 |
| CI 集成 | 需要 webhook 接收 |
| 自动返工 | 需要 Reviewer AI |
| 质量门禁 | 需要聚合规则引擎 |
| 截图文件上传 | v0.2 只保存文本型报告和链接 |

## 3. 数据模型设计

### 3.1 TestRun 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK AUTO | 自增主键 |
| `task_id` | INTEGER FK → Task.id | 关联任务 |
| `project_id` | INTEGER FK → Project.id | 关联项目 |
| `test_type` | TEXT NOT NULL | `playwright` / `api` / `unit` / `integration` |
| `runner` | TEXT NOT NULL | `playwright_mcp` / `manual` / `ci` |
| `status` | TEXT NOT NULL | `queued` / `running` / `passed` / `failed` / `canceled` |
| `scenario_name` | TEXT | 测试场景名称（如 "用户登录后查看仪表盘"） |
| `target_url` | TEXT | 测试目标 URL |
| `branch` | TEXT | 测试时分支名 |
| `commit_sha` | TEXT | 测试时 commit hash |
| `started_at` | DATETIME | 开始时间 |
| `finished_at` | DATETIME | 结束时间 |
| `duration_ms` | INTEGER | 耗时（毫秒） |
| `summary` | TEXT | 测试结果摘要 |
| `console_error_count` | INTEGER DEFAULT 0 | 浏览器控制台错误数 |
| `network_failure_count` | INTEGER DEFAULT 0 | Network 请求失败数 |
| `failed_step` | TEXT | 失败步骤描述 |
| `raw_report_json` | TEXT | 原始报告 JSON（Playwright JSON 格式） |
| `created_at` | DATETIME DEFAULT now | 创建时间 |
| `updated_at` | DATETIME | 最后更新时间 |

### 3.2 TaskArtifact 新增类型

| 类型 | 说明 | 存储方式 |
|------|------|----------|
| `playwright_report` | Playwright JSON 报告文本 | sqlite |
| `playwright_trace` | Trace 查看链接或路径 | sqlite + file_path |
| `screenshot` | 截图文件路径或 base64 | file_path |
| `browser_console` | 控制台日志文本 | sqlite |
| `network_log` | Network 请求日志文本 | sqlite |
| `test_video` | 测试录屏路径 | file_path |

> **说明**：v0.2 阶段仅存储文本型测试产物（playwright_report、browser_console、network_log）。截图/trace/video 的 file_path 字段预留，不做文件上传实现。

### 3.3 TaskEvent 新增类型

| event_type | 触发时机 |
|-----------|----------|
| `test_run_created` | 创建 TestRun |
| `test_run_updated` | 更新 TestRun 状态 |
| `test_run_passed` | status → passed |
| `test_run_failed` | status → failed |
| `test_artifact_uploaded` | 上传测试相关 Artifact |

### 3.4 ER 关系

```
Task 1──N TestRun
TestRun 1──N TaskArtifact (via task_id + artifact_type prefix)
Task 1──N TaskEvent (含 test_run_* 类型)
```

## 4. API 设计

全部返回 `ApiEnvelope<T>`。前缀 `/api`。

### 4.1 TestRun

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/tasks/{id}/test-runs` | 创建 TestRun（body 含 test_type、runner、scenario_name、target_url、branch、commit_sha） |
| `GET` | `/api/tasks/{id}/test-runs` | TestRun 列表（按 created_at DESC） |
| `GET` | `/api/tasks/{id}/test-runs/{run_id}` | TestRun 详情 |
| `PATCH` | `/api/tasks/{id}/test-runs/{run_id}` | 更新 TestRun（状态、耗时、错误数、失败步骤、摘要等） |
| `DELETE` | `/api/tasks/{id}/test-runs/{run_id}` | 删除 TestRun（仅 queued 可删） |

### 4.2 TestRun 产物

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/tasks/{id}/test-runs/{run_id}/artifacts` | 上传测试产物（artifact_type + content） |
| `GET` | `/api/tasks/{id}/test-runs/{run_id}/artifacts` | 测试产物列表 |

### 4.3 请求/响应示例

**创建 TestRun：**
```json
POST /api/tasks/1/test-runs
{
  "test_type": "playwright",
  "runner": "playwright_mcp",
  "scenario_name": "用户登录后查看仪表盘",
  "target_url": "http://127.0.0.1:9700",
  "branch": "fix/ui-polish",
  "commit_sha": "10ab622"
}
→ 201 { "data": { "id": 1, "status": "queued", ... } }
```

**更新 TestRun（提交结果）：**
```json
PATCH /api/tasks/1/test-runs/1
{
  "status": "passed",
  "started_at": "2025-07-16T12:00:00Z",
  "finished_at": "2025-07-16T12:05:00Z",
  "duration_ms": 300000,
  "summary": "全部 6 个场景通过",
  "console_error_count": 0,
  "network_failure_count": 0
}
→ 200 { "data": { "id": 1, "status": "passed", ... } }
```

**更新 TestRun（失败）：**
```json
PATCH /api/tasks/1/test-runs/1
{
  "status": "failed",
  "duration_ms": 240000,
  "summary": "3/6 通过，1 个场景失败",
  "console_error_count": 2,
  "network_failure_count": 1,
  "failed_step": "ProjectConfig: 更新项目配置后页面未刷新"
}
→ 200 { "data": { "id": 1, "status": "failed", ... } }
```

## 6. 状态和审计

### TestRun 状态机

```
queued → running → passed
                → failed
                → canceled
```

### TaskEvent 联动规则

| 操作 | 写入 Event |
|------|-----------|
| 创建 TestRun | `test_run_created`（status=queued） |
| 开始运行 | `test_run_updated`（status=running） |
| 测试通过 | `test_run_passed` |
| 测试失败 | `test_run_failed` |
| 上传测试产物 | `test_artifact_uploaded` |

## 7. 安全边界

| 规则 | 说明 |
|------|------|
| 不自动执行 Playwright | v0.2 只做结果记录和展示，不启动浏览器进程 |
| 不启动浏览器进程 | 无 subprocess / puppeteer 调用 |
| 不执行 shell | 所有测试结果通过 API 提交 |
| 不访问 Project.root_path | 无文件系统操作 |
| 不接收真实 token | TestRun 无 token/secret 字段 |
| 不接入外部 CI | 仅 manual / playwright_mcp runner |
| 不直接影响部署 | 测试结果不影响自动部署 |
| 文本型展示 | 测试报告仅做文本展示，不渲染 HTML |
| 截图文件限制 | v0.2 只存文件路径/链接，不实现文件上传 |
| archived 保护 | archived Task 拒绝所有 TestRun 写入 |

## 设计决策说明

### 为什么不复用 ReviewRecord？

ReviewRecord 是人工审查记录，核心语义是"人做的决策"。TestRun 是自动化测试结果，频率更高、字段不同、生命周期独立。拆开更清晰。

### 为什么 TestRun 也需要 project_id？

虽然 TestRun 通过 task_id → project_id 可间接获取，但测试结果经常需要跨任务查询（如"项目 X 的最新测试通过率"）。冗余 project_id 避免了深层 JOIN。

### 为什么 raw_report_json 不分表？

Playwright 原始 JSON 报告结构复杂且不固定。直接存 TEXT 字段让前端/外部工具按需解析，避免模型频繁变更。
